The Supabase India Block

Supabase was blocked in India. The impact was immediate across multiple production projects:

Knight Medicare — CBT therapy platform, production data layer. Open Permit — civic intelligence platform, core database. Adventurers Guild — developer marketplace, user data and quest storage. AfA Resource Chatbot — RAG system, vector storage (pgvector). Project Slingshot UK — data pipeline, 12+ API sources.

When your primary database provider gets blocked in your country, you learn very quickly how important infrastructure abstraction is.

The Three Options

Option A: Wait It Out

Hope the block reverses. Verdict: unacceptable for production platforms. No control over timeline.

Option B: Yotta Self-Hosted ✓

Migrate to Yotta Data Centre (Panvel). Self-hosted Supabase on 4 vCPU/16GB KVM VM. 63%+ cost reduction. Full data sovereignty.

Option C: Alternative BaaS

Firebase (no PostgreSQL), PlanetScale (no pgvector), Neon (works for new projects but complex migration for existing ones).

Migration Scope: 29 Services

Migration Inventory: ├── Knight Medicare: Auth, Database, Realtime, Storage ├── Open Permit: Auth, Database, Edge Functions ├── Project Slingshot: Database, Scheduled Functions ├── AfA Chatbot: pgvector, Database └── [7 more projects]: Various Supabase services Total: 29 services across 11 projects

Abstraction Layer Strategy

The key decision: don't just migrate, abstract. Claude Code helped design database abstraction layers so swapping from Supabase Cloud to self-hosted PostgreSQL requires changing adapter code, not business logic. The pattern: define generic interface → implement Supabase adapter (current) → implement self-hosted adapter (target) → swap via environment variable → run both in parallel during validation.

This means the next infrastructure disruption doesn't require rewriting business logic across 11 projects.

Yotta Data Center Visit (March 11)

Visited the Yotta Data Center in Panvel. Evaluated physical security, network connectivity/latency, VM provisioning tools, support SLAs, and partnership possibilities. Confirmed as the right choice for Indian sovereign hosting — single-digit millisecond latency to Mumbai, reasonable nationwide.

SMS Provider: Dual Approach

Knight Medicare and other projects need SMS for auth/notifications. Solution: MSG91 for India (best domestic rates, DLT compliance built-in) + Plivo for international. Dual routing based on user geography. Evaluated through Claude Code comparing pricing, API capabilities, delivery rates, and DLT requirements.

Key Lessons

Never lock into a single provider. Abstraction layers aren't overengineering — they're insurance. Self-hosting is cheaper at scale — for 11 projects with 29 services, a single VM saves 63%+ vs. managed services. Indian sovereign cloud matters — latency advantage, reliability, and compliance. Infrastructure planning is product planning — reliable infra means therapy sessions don't drop, objection letters generate on time, quest assignments don't fail.
